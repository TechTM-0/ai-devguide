# フロー（業務・処理の流れ）の全体像と現状

このドキュメントは「このフレームワークのフローが**実際にどう動くか**」と「**まだ決まっていないこと**」を正直に分けて書いたもの。
README の図は"思考の順序"を1枚で描いているだけで、時間的な遷移・ステータス・入れ子・Issue管理は描かれていない。その穴を埋めるのが本書。

> 用語は `[日本語の意味（english）]` 形式で併記する（RULES.md §7 言語・表現規律）。

---

## 1. 今この瞬間、実際に動いているフロー（実装済み）

機械（フック）が本当にやっているのは、次の4点だけ。

| いつ | 何が起きる | 何をしない |
|---|---|---|
| セッション開始 | `current.md`（今の状態）を文脈に注入する | フェーズの自動判定・自動再開はしない（Claude が読んで判断） |
| 指示を送るたび | 「方針/手法/タスク構成を変える必要があるか」を1行で表示させる | 変更の中身は強制しない（人/AIの判断） |
| 編集の前 | `log.md`（履歴）の関連箇所を確認させる | 確認したかの検証はしない |
| 編集の後 | `log.md` に決定を追記し `current.md` を更新させる | 内容の質はチェックしない |

**重要な事実：フェーズの遷移そのものは「自動」ではない。** いま何フェーズかは `current.md` の「現在地」欄に Claude が**手で書く**文字列にすぎない。
機械は「記録しろ」と促すだけで、「問題定義は終わった、次は構造設計だ」という**判定はしていない**。

```
[実際の動き]
セッション開始 → current.md を注入（機械）
   ↓
Claude が「現在地」を読んで、人間と相談しながら作業（ここは全部 人＋AI の判断）
   ↓
編集の前後で「履歴を見ろ」「記録しろ」と促される（機械）
   ↓
Claude が log.md に追記・current.md を上書き（手作業）
```

つまり今のフローは **「順序の手本（README の図）＋ 記録を促すフック」** であって、
**状態を持って自動で遷移する仕組みではない。**

## 4. これからの方向性（未決定・要判断）

本質的な分岐は1つ：**狙う対象はどちらか。**

- **(a) 使い捨て含む全プロジェクトを狙う** → 設定ゼロ・自動適用が効く**ローカル完結**が有利。穴 A〜G は軽量に（current.md にステータス語彙を足す程度）埋める。
- **(b) ゴールのある本気のプロジェクトを狙う** → **GitHub 活用**が有利。
  - `[課題（Issue）]` ＝ 問題定義＋完了条件、`[下位課題（sub-issue）]` で入れ子（穴 D, F を解消）
  - `[変更提案（Pull Request）]` ＝ 実装＋レビュー（①②を解消）
  - `[タスク管理ボード（Project Board）]` ＝ カラムでステータス可視化（穴 A を解消）
  - コメント＋コミット ＝ 改ざんしにくい決定履歴（④を解消）
- **(c) ハイブリッド（推奨）** → 判断の規律はローカルのフック、プロジェクト規模の構造は GitHub。

> この (a)/(b)/(c) が決まると、穴 A〜G の埋め方が連動して決まる。**まずここを決めるのが先決。**

## 5. 役割分担の全体マップ（ハイブリッド案・提案段階）

> 目的（4課題）は不変。これは「目的を果たす手段」としての役割分担。**大半は未実装の提案**であり、現状で動いているのは「ローカルフック＝促し」のみ（§1）。

フロー全体を通して、各場面で誰が何を担当するか：

| フェーズ／場面 | 人間＝**決める** | AI＝**考える・作る** | GitHub＝**残す・見せる** | ローカルフック＝**促す** |
|---|---|---|---|---|
| 0. ゴール・問題定義 | 問題と最終ゴールを与える | 問題定義・入出力・責任分割の叩き台 | 親 `[課題（Issue）]`／`[一里塚（Milestone）]`＋完了条件 | 問題定義・成功基準の有無を促す |
| 1. ロジック設計（構造） | 分割・責任分離を承認 | 構造案・工程分割・サブ課題への分解 | 子Issueで起票（入れ子）＋フェーズのラベル | Logic Review 項目を促す |
| 2. アルゴリズム選定 | 選定を承認（方針はGO必須） | 複数手法の比較・選定理由の原案 | 比較・決定をIssue/PRコメントに記録 | 比較・理由記録を促す |
| 3. 実装 | レビュー・マージ承認 | コードを書く | `[変更提案（Pull Request）]` で差分・レビュー・マージ | 編集前後で履歴確認・記録を促す |
| 4. 評価 | 完了をサインオフ | 完了条件への検証結果（暫定） | 完了条件チェック・ボードを完了へ | Evaluator Design（4軸）を促す |
| 5. ステータス管理・遷移 | 優先順位・WIP上限を決める | ステータス更新操作・次の一手の提案 | ボードのカラムで現在地を可視化 | （GitHub未使用時）current.md更新を促す |
| 6. 上流変更・Issue波及 | 方針変更のGO | 影響する下流Issueの洗い出し・要見直し提案 | 親子リンクでまたいだ波及を可視化 | 上流変更の検知を促す（穴B-4） |
| 7. 最適化ループ／メタ改善 | 問題設定・評価基準を疑う判断のGO | 原因特定→改善案 | 再オープンIssue・議論履歴 | 「基準を疑え」を促す |

### 一言でいう境界

- **AI ＝「中身（意味）」**：考える・分解する・作る・暫定判断する。
- **GitHub ＝「器（形）」**：入れ物・親子構造・ステータス表示・改ざんしにくい履歴。
- **人間 ＝「決定」**：ゴール・優先順位・方針GO・完了サインオフ。
- **ローカルフック ＝「促し」**：GitHub外でも規律を効かせる保険。

根拠：log 06:36「機械（フック）は生成・駆動できない／意味のある作業はAI／最終決定は人間」に、GitHub を"形と履歴の器"として加えたもの。AIは器を作れるが決定の信頼性が無いため、サインオフは人間が担う。
